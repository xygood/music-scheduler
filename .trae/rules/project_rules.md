# 项目开发规范 (Project Constitution)

## 核心原则

### I. 规范优先 (Spec-First)
- **所有功能开发必须先写规格文档，后写代码**
- 规格文档需包含：用户故事、验收标准、技术约束
- 代码必须为规格服务，而非规格为代码服务

### II. 结构化开发流程
- **阶段 1**: 需求规格 (Specify) - 明确"做什么"
- **阶段 2**: 技术方案 (Plan) - 明确"怎么做"
- **阶段 3**: 任务拆解 (Tasks) - 明确"具体步骤"
- **阶段 4**: 编码实现 (Implement) - 按图施工

### III. 文档即代码
- 每个开发阶段必须产出标准格式的 Markdown 文档
- 文档存放在 `specs/` 目录，版本化管理
- 文档与代码同步更新，禁止文档滞后

### IV. 质量门禁
- 核心功能必须有单元测试覆盖
- 代码需通过静态检查 (lint/typecheck)
- 提交前需自测通过

### V. 简洁优先
- 遵循 YAGNI 原则，不预设未来需求
- 优先使用原生能力，减少外部依赖
- 代码自解释，注释说明"为什么"而非"做什么"

## 技术栈约束

- **语言**: 根据项目类型确定
- **风格**: 遵循项目现有代码风格
- **测试**: 新功能必须包含测试用例

## 开发工作流

1. 在 `specs/` 创建需求文档
2. 与 AI 讨论并完善方案
3. 按任务列表逐步实现
4. 自测并修复问题
5. 提交代码


# 代码修改黄金法则
1.  **最小改动原则**：在进行任何修改前，先分析和解释现有代码的功能。除非必要，否则不得改动无关的函数、变量或逻辑。
2.  **保持原有架构**：新增功能时，优先遵循项目现有的设计模式和代码组织结构，除非我明确要求重构。
3.  **完整回归**：修改A功能后，必须检查并确保所有依赖或调用A功能的B、C等模块不受影响，并在注释中说明检查结果。
